1. INTRODUCTION

1.1 Project Overview
This project implements a comprehensive Hospital Management and Patient Management 
System using a hybrid database approach combining SQL (MySQL) and NoSQL (MongoDB) 
technologies. The system is designed to handle structured, semi-structured, and 
unstructured data to meet real-world healthcare application needs.

1.2 Objectives
- Design and implement a scalable healthcare database system
- Leverage SQL for transactional and ACID-compliant data (structured data)
- Utilize NoSQL for flexible storage of medical records, imaging, and logs
- Create efficient integration between SQL and NoSQL systems
- Implement comprehensive CRUD operations and analytical queries
- Provide practical insights into hybrid database architecture

1.3 Domain Selection: Healthcare System
Healthcare was chosen as the domain because:
- Complex structured data requirements (patient records, appointments, billing)
- Need for unstructured data storage (medical notes, imaging reports)
- High availability and data integrity requirements
- Real-world applicability and relevance
- Opportunity to demonstrate both transactional and analytical capabilities

4.2 Key SQL Queries

Doctor Workload Report:
SELECT d.DoctorID, CONCAT(s.FirstName, ' ', s.LastName) AS DoctorName,
       d.Specialization, COUNT(a.AppointmentID) AS TotalAppointments,
       SUM(CASE WHEN a.Status = 'Completed' THEN 1 ELSE 0 END) AS CompletedAppointments
FROM Doctors d
LEFT JOIN Staff s ON d.StaffID = s.StaffID
LEFT JOIN Appointments a ON d.DoctorID = a.DoctorID
GROUP BY d.DoctorID, s.FirstName, s.LastName, d.Specialization
ORDER BY TotalAppointments DESC;

Patient Billing Summary:
SELECT p.PatientID, CONCAT(p.FirstName, ' ', p.LastName) AS PatientName,
       COUNT(b.BillingID) AS TotalTransactions,
       SUM(b.TotalAmount) AS TotalBilled,
       SUM(CASE WHEN b.PaymentStatus = 'Paid' THEN b.TotalAmount ELSE 0 END) 
           AS TotalPaid,
       SUM(CASE WHEN b.PaymentStatus IN ('Pending', 'Partial') THEN b.TotalAmount 
           ELSE 0 END) AS OutstandingBalance
FROM Patients p
LEFT JOIN Billing b ON p.PatientID = b.PatientID
GROUP BY p.PatientID, p.FirstName, p.LastName
HAVING TotalBilled > 0
ORDER BY OutstandingBalance DESC;

Department Performance:
SELECT d.DepartmentName,
       COUNT(DISTINCT a.PatientID) AS TotalPatients,
       COUNT(a.AppointmentID) AS TotalAppointments
FROM Departments d
LEFT JOIN Appointments a ON d.DepartmentID = a.DepartmentID
GROUP BY d.DepartmentID, d.DepartmentName
ORDER BY TotalAppointments DESC;

4.3 Key MongoDB Queries

Medical Condition Analysis:
db.medicalRecords.aggregate([
    { $unwind: "$symptoms" },
    { $group: {
        _id: "$symptoms",
        count: { $sum: 1 },
        patients: { $push: "$patientId" }
    }},
    { $sort: { count: -1 } },
    { $limit: 10 }
]);

Patient Feedback Analytics:
db.patientFeedback.aggregate([
    { $group: {
        _id: "$departmentName",
        averageRating: { $avg: "$rating" },
        totalReviews: { $sum: 1 },
        recommendationRate: { $avg: { $cond: ["$recommendation", 1, 0] } }
    }},
    { $sort: { averageRating: -1 } }
]);

System Activity Trends:
db.systemLogs.aggregate([
    { $group: {
        _id: { $dateToString: { format: "%Y-%m-%d", date: "$timestamp" } },
        count: { $sum: 1 },
        successCount: { $sum: { $cond: [{ $eq: ["$status", "Success"] }, 1, 0] } },
        failureCount: { $sum: { $cond: [{ $eq: ["$status", "Failed"] }, 1, 0] } }
    }},
    { $sort: { _id: -1 } }
]);

6. CHALLENGES & SOLUTIONS
================================================================================

6.1 Challenges Encountered

Challenge 1: Data Consistency Between SQL and NoSQL
Problem: Ensuring data consistency when same entity is referenced in both databases
Solution: Used PatientID as reliable reference, maintained synchronization through 
         timestamps and application-level logic, implemented transactional guarantees 
         for critical operations

Challenge 2: Complex Queries Across Databases
Problem: Joining data from SQL and NoSQL sources is not straightforward
Solution: Implemented application layer in Python to orchestrate queries across both 
         databases, retrieved data from each source separately and merged in memory

Challenge 3: Schema Design Trade-offs
Problem: Deciding which data should be structured (SQL) vs unstructured (NoSQL)
Solution: SQL for transactional data requiring ACID compliance, NoSQL for flexible 
         medical records and audit logs, clear separation of concerns

Challenge 4: Integration Testing
Problem: Testing interactions between SQL and NoSQL components
Solution: Created comprehensive test cases, used Docker for consistent database 
         environments, implemented logging for debugging

Challenge 5: Performance Optimization
Problem: Initial queries were slow with large datasets
Solution: Added strategic indexes, used aggregation pipelines efficiently, 
         implemented caching where applicable

6.2 Solutions Implemented

1. ACID Transactions:
   - Implemented for critical operations in SQL
   - Document-level atomicity in MongoDB
   - Application-level transaction coordination

2. Data Synchronization:
   - Timestamp-based tracking
   - Audit logs for changes
   - Eventual consistency model

3. Error Handling:
   - Try-catch blocks in Python
   - Connection retry logic
   - Graceful error messages

4. Security:
   - Password protection for databases
   - Input validation
   - SQL injection prevention with parameterized queries
